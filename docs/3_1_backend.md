# 3.1. Серверна частина (Backend)

**У процесі розробки системи управління задачами** було обрано модульно‑шарову архітектуру, що передбачає чітке розділення функціональних частин системи на окремі логічні блоки. Система поділена на серверну (backend) та клієнтську (frontend) підсистеми, кожна з яких розміщена у власному Git‑репозиторії. Така організація забезпечує **ізоляцію інтерфейсу користувача від бізнес‑логіки**, спрощує **підтримку**, підвищує **масштабованість** та дозволяє **незалежне CI/CD** для кожного модуля.

### 3.1.1. Налаштування середовища (Node.js, TypeScript, Fastify)

Серверна частина реалізована на **Node.js** **LTS** у поєднанні з **TypeScript**. Статична типізація, яку пропонує TypeScript, дозволяє виявляти логічні та типові помилки ще під час компіляції, що значно підвищує надійність коду й полегшує його подальше обслуговування.  

Структура коду організована у вигляді кореневої директорії `src`, де піддиректорії відповідають за окремі аспекти функціональності:
- `app.ts` – точка входу, створює інстанс **Fastify** і підключає базові плагіни (CORS, Helmet).
- `plugins/` – підключення сторонніх пакетів: **Prisma** (ORM), **JWT** (автентифікація) та інші.
- `routes/` – оголошення HTTP‑маршрутів застосунку.
- `controllers/` – обробка вхідних запитів, підготовка відповіді.
- `services/` – бізнес‑логіка, що координує роботу контролерів і репозиторіїв.
- `repositories/` – взаємодія з базою даних через Prisma.
- `middlewares/` – глобальні та роут‑специфічні проміжні обробники (автентифікація, обробка помилок).
- `schemas/` – схеми валідації вхідних даних за допомогою **Zod**.

У файлі `package.json` визначено скрипти, які автоматизують розробку:
- `dev` – запуск у режимі hot‑reload за допомогою `tsx`.
- `build` – трансляція TypeScript у папку `dist`.
- `start` – продакшн‑запуск готового коду.
- `prisma:generate` та `prisma:migrate` – генерація типів Prisma і застосування міграцій до PostgreSQL.

**Fastify** обрано за його **микросервісну модель**, **високу продуктивність** та **плагін‑орієнтовану архітектуру**. Плагіни легко підключаються, що дозволяє швидко додавати функціональність (наприклад, CORS, Helmet, аутентифікацію). Нижче наведено стислий фрагмент ініціалізації сервера:
```ts
import Fastify from 'fastify';
import cors from '@fastify/cors';
import helmet from '@fastify/helmet';

const app = Fastify({ logger: true });
await app.register(cors, { origin: process.env.CORS_ORIGIN, credentials: true });
await app.register(helmet);
// подальші плагіни та реєстрація роутів
```
Цей код демонструє лише базове підключення middleware, після чого слідує реєстрація всіх роутів і запуск сервера.

### 3.1.2. Prisma та інфологічна схема

Для зберігання даних використано **PostgreSQL**, яка забезпечує **транзакційну цілісність**, **розширені типи** та можливість масштабування. На рівні ORM використовується **Prisma**, що генерує **типізовані клієнти** на основі схеми даних, що суттєво полегшує розробку та знижує кількість SQL‑помилок.

У файлі `prisma/schema.prisma` описано чотири головні сутності: `User`, `Group`, `Task` та `TaskDependency`. Кожна сутність має унікальний `uuid`, поля часу створення/оновлення та необхідні зв’язки. Наприклад, сутність **User** містить електронну пошту, геш пароля, роль у системі та зв’язки з групами і задачами.  
```prisma
model User {
  id           String   @id @default(uuid())
  email        String   @unique
  name         String?
  passwordHash String
  role         Role     @default(USER)
  groups       Group[]  @relation("GroupMembers")
  tasks        Task[]   @relation("Assignee")
}
```
Міграції створюються командами `prisma migrate dev` та `prisma generate`. Це гарантує, що **структура бази даних завжди синхронізована** з кодовою моделлю, а типи, створені Prisma, доступні у всьому проєкті, що підвищує **безпеку** та **зручність розробки**.

### 3.1.3. Архітектурний патерн Controller‑Service‑Repository

Для забезпечення **чистоти коду** та **полегшення тестування** застосовано патерн **Controller‑Service‑Repository**.  
- **Контролери** відповідають лише за прийом HTTP‑запиту, делегування його сервісу і формування відповіді.  
- **Сервіси** містять **бізнес‑логіку** (перевірка прав, трансформація даних, оркестрування викликів кількох репозиторіїв).  
- **Репозиторії** взаємодіють безпосередньо з **Prisma‑клієнтом** і виконують CRUD‑операції.

Такий підхід дозволяє **модульно тестувати** кожен шар: контролери можна мокати, сервіси протестувати в ізоляції, а репозиторії – у інтеграції з тестовою БД.  
Для валідації вхідних даних використано **Zod**. Оскільки Zod **генерує типи** під час компіляції, всі вхідні дані суворо типізовані, що знижує ризик неочікуваних помилок. Приклад схеми створення задачі:
```ts
export const taskCreateSchema = z.object({
  title: z.string().min(1),
  description: z.string().optional(),
  groupId: z.string().uuid(),
});
```
Контролери отримують уже валідовані дані і безпечно передають їх у сервіс.

### 3.1.4. Безпека та реальний час

#### JWT‑автентифікація та Refresh‑token rotation
Для захисту REST‑API використано **JWT**. Токени доступу (`accessToken`) мають короткий термін дії (≈15 хв.) і передаються у заголовку `Authorization: Bearer <token>`. Довший термін дії мають **refresh‑токени**, які зберігаються у **HTTP‑only, Secure cookie**. При запиті на оновлення токену сервер перевіряє актуальність refresh‑токену, генерує нову пару токенів і позначає старий як **revoked**. Така **rotation** запобігає повторному використанню компрометованих токенів і відповідає рекомендаціям OWASP.

#### Role‑Based Access Control (RBAC)
Контроль доступу реалізовано за допомогою двох middleware:
- `authGuard` – розшифровує JWT, отримує інформацію про користувача і додає її до об’єкта запиту.
- `roleGuard` – приймає список дозволених ролей і перевіряє, чи входить роль користувача у цей список.  
Це дозволяє **гнучко керувати правами** (наприклад, лише адміністратори можуть видаляти задачі), не дублюючи логіку в кожному контролері.

#### Socket.io для обміну у реальному часі
Для синхронізації стану Kanban‑дошки використано **Socket.io**. Після встановлення WebSocket‑з’єднання клієнт передає JWT у параметрі `auth`. Сервер аутентифікує користувача, після чого додає сокет до **кімнат**, що відповідають групам (`group:<id>`). При створенні, оновленні або видаленні задачі сервер генерує відповідну подію (`task:created`, `task:updated`, `task:deleted`) і надсилає її лише до потрібних кімнат, що **мінімізує мережеве навантаження**. На фронтенді підписка на ці події оновлює стан, збережений у **Zustand**, і користувач миттєво бачить зміни без додаткових запитів до API.

### Діаграма архітектури
```mermaid
graph LR
    subgraph Client
        UI[React UI] --> WS[WebSocket (Socket.io)]
        UI --> HTTP[REST API Calls]
    end
    subgraph Server
        Fastify[Fastify] --> Router[Routes]
        Router --> Controllers[Controllers]
        Controllers --> Services[Services]
        Services --> Repos[Repositories]
        Repos --> Prisma[Prisma ORM]
        Prisma --> PG[PostgreSQL]
        Fastify --> WS
    end
    WS -->|real‑time events| UI
    HTTP -->|CRUD| UI
```
Діаграма ілюструє взаємодію клієнтської частини (React UI) з сервером: стандартні запити здійснюються через **REST‑API**, а сповіщення у реальному часі – через **WebSocket** (Socket.io). Серверна частина, побудована на **Fastify**, обробляє запити за патерном **Controller‑Service‑Repository**, а дані зберігаються у **PostgreSQL** через **Prisma**.
