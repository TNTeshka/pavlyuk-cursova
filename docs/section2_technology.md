# Огляд технологій та засобів розробки
`
## 1. Вступ

Сучасний веб‑проєкт вимагає швидкої розробки, високої продуктивності, зручності підтримки, масштабованості та можливості кастомізації інтерфейсу під різні вимоги користувачів. У цьому розділі детально розглядається технологічний стек, обраний для розробки системи управління задачами **Task Manager**, а також проводиться порівняльний аналіз з альтернативними рішеннями. Оцінка базується на практичному досвіді розробки, результатах бенчмарків, а також на відгуках спільноти.

## 2. Перелік технологій та їх роль у проєкті
`

| **Vercel / Netlify** | Хостинг/CI‑CD, автоматичний деплой, Edge Functions. | Автоматичний preview‑доп, безкоштовний тариф для малих проєктів, просте підключення Git. | AWS Amplify, Firebase Hosting |
| **Jest + React Testing Library** | Юніт‑тести та інте‑тести компонентів, перевірка логіки. | Підтримка TypeScript, швидка інтеграція, хороші мок‑фреймворки. | Mocha, Cypress |
| **ESLint + Prettier** | Лінтинг та автоформатування коду. | Строгі правила, автоматичне форматування під час commit (Husky). | TSLint (застарілий), StandardJS |

## 3. Порівняльний аналіз з альтернативами

### 3.1 Фронтенд‑фреймворки

| Фреймворк | Переваги | Недоліки | Примітка щодо використання в проєкті |
|---|---|---|---|
| **React** | Компонентність, Virtual DOM, реактивність, головний вибір у індустрії, широка екосистема (React Query, Redux Toolkit). | Потрібна додаткова бібліотека для управління станом (наприклад, Redux), не має «з коробки» MVC‑патерну. | Обрано через готовність до TypeScript, легкість інтеграції з Tailwind та Vite. |
| **Vue 3** | Прогресивна архітектура, простіший шаблонний синтаксис, Composition API, легкої міграції. | Менша кількість корпоративних бібліотек у порівнянні з React, менш розповсюджений у великих компаніях. | Можливий варіант, проте команда вже має досвід з React. |
| **Angular** | Повний фреймворк (DI, RxJS, шаблони), інструментарій CLI, TypeScript «з коробки». | Важка крива навчання, великі boilerplate‑файли, обмежена кастомізація UI без сторонніх бібліотек. | Не підходить через надлишкову складність для MVP‑проєкту. |
| **Svelte** | Надзвичайно маленький розмір бандла, автоматичне оновлення реактивних змінних. | Менша спільнота, обмежений набір готових компонентів, не так поширений у великих проектах. | Перспективний, але надлишковий ризик нестабільності. |

### 3.2 Інструменти збірки

| Інструмент | Переваги | Недоліки | Коментар |
|---|---|---|---|
| **Vite** | Швидка HMR (мс), zero‑config, підтримка ES‑модулів, інтеграція з React‑Fast Refresh. | При дуже великій кількості складних плагінів може стати менш гнучким, ніж Webpack. | Обрано за простоту і швидкість роботи. |
| **Webpack** | Надзвичайна гнучкість, підтримка великої кількості loaders та plugins. | Складна конфігурація, довгий час збірки, тяжка підтримка. | Не підходить для швидкої розробки MVP. |
| **Parcel** | Zero‑config, автоматичне розпізнавання файлів. | Менш контроль над оптимізацією, обмежений екосистемою плагінів. | Підійде лише для дуже простих прототипів. |
| **Snowpack** | Попередньо компільований bundler, швидка розробка. | Схожі обмеження як у Vite, менша популярність. | Аналог Vite, проте Vite отримав ширшу підтримку. |

### 3.3 CSS‑фреймворки

| Фреймворк | Переваги | Недоліки |
|---|---|---|
| **Tailwind CSS** | Утилітарний підхід, purge‑оптимізація, підтримка dark/light тем, швидка прототипізація. | Потрібно вивчати велику кількість класів, можливе «фрагментація» HTML. |
| **Sass/SCSS** | Можливість створення змінних, міксінів, функцій. | Потрібно писати власні CSS‑правила, збільшує розмір результ. |
| **Bootstrap** | Готові UI‑компоненти, сітка, підтримка тем. | Менше гнучкості в кастомізації, обмеження дизайну. |
| **styled‑components / Emotion** | CSS‑in‑JS, динамічне стилювання. | Підвищений runtime‑overhead, збільшує bundle. |

### 3.4 ORM‑шари

| ORM | Переваги | Недоліки |
|---|---|---|
| **Prisma** | Type‑safe клієнт, автоматична генерація типів, прості міграції, хороша документація. | Працює лише з певними СУБД (PostgreSQL, MySQL, SQLite, MongoDB), обмежена підтримка складних запитів без raw SQL. |
| **TypeORM** | Підтримка декораторів, активна підтримка типів, широкі можливості. | Менша стабільність у надходжених версіях, складний API, часто виникають баги з lazy relations. |
| **Sequelize** | Підтримка багатьох баз, легка модель ES6‑класів. | Не генерує типи автоматично, потребує явно написаних інтерфейсів. |
| **MikroORM** | Підтримка TypeScript, діапазон СУБД, легка міграція. | Менша спільнота, менше документації. |

### 3.5 Маршрутизація SPA

| Бібліотека | Переваги | Недоліки |
|---|---|---|
| **React Router** | Декларативне оголошення роутів, підтримка nested routes, стабільність. | Не підтримує SSR «з коробки», треба вручну налаштовувати. |
| **Reach Router** | Легша бібліотека, проста API, менш громіздка. | Припинив підтримку, не оновлюється. |
| **Next.js routing** | Файлова система роутів, підтримка SSR/SSG. | Переваги для сервер‑рендера, зайва складність для чисто SPA. |
| **Wouter** | Мікробібліотека, 1 KB, простий API. | Обмежений функціонал, відсутня підтримка захищених роутерів. |

### 3.6 Валідизація даних

| Бібліотека | Переваги | Недоліки |
|---|---|---|
| **Zod** | Автогенерація типів, простий схематичний опис, підтримка async‑валидації. | Менше розповсюджена в порівнянні з Joi. |
| **Joi** | Широка функціональність, багаті валідації. | Не генерує TypeScript‑типи, громіздка. |
| **Yup** | Перш за все для форм, схоже на Joi. | Потрібно вручну писати типи, відсутня підтримка TypeScript‑generation. |
| **Ajv** | JSON‑Schema валідація, швидкість. | Потребує окремих схем, не інтегрується безпосередньо в TS. |

## 4. Глибокий аналіз обраного стеку

### 4.1 React + TypeScript
- **Компонентність** дозволяє розділити UI на незалежні, пере‑використовувані блоки (`Button`, `Header`, `Sidebar`).
- **Hooks** (`useState`, `useEffect`, `useMemo`, `useCallback`) забезпечують декларативне управління станом і побічними ефектами, що полегшує підтримку.
- **React Fast Refresh** у Vite забезпечує миттєве оновлення без втрати стану компоненту.
- **TypeScript** інтегрує типи **Prisma**, **Zod**, що підвищує надійність API‑взаємодії.

### 4.2 Vite
- **Instant Server Start** (меньше секунди) та **Lightning‑fast HMR** (події оновлення в <100 мс).
- **Rollup‑based prod‑bundle** з tree‑shaking, що дає фінальний розмір <150 KB (gzip).
- **Built‑in support** for JSX, TypeScript, PostCSS (Tailwind), що спрощує налаштування.

### 4.3 Tailwind CSS
- **Утилітарний підхід** дозволяє уникнути каскадних пере‑визначень, підвищує консистентність UI.
- **PurgeCSS** в продакшн‑режимі залишає лише використані класи → **35 KB gzipped**.
- **Dark/Light Themes** реалізовані через `dark:` префікси та кастомні CSS‑змінні (`--color-bg`, `--color-primary`).
- **Градієнтний Header**: у файлі `custom.css` додано градієнт `linear‑gradient(180deg, rgba(30,30,30,0.75) 0%, rgba(30,30,30,0.45) 50%, rgba(30,30,30,0.2) 100%)` + `backdrop-filter: blur(10px);`.

### 4.4 Prisma
- **Prisma Client** генерує TypeScript‑типи для схеми, що дозволяє писати тип‑безпечні запити:
```ts
const tasks = await prisma.task.findMany({ where: { status: 'TODO' } });
```
- **Migrations** (`prisma migrate dev`) забезпечують контроль змін структури БД та їх повторюваність.
- **Performance**: запити компілюються в parametrized SQL, використовуючи prepared statements.

### 4.5 React‑Router
- **Declarative routes** у `App.tsx` спрощують розширення (наприклад, додаємо нову сторінку `Reports`).
- **Protected route** – компонент `Protected` перевіряє JWT‑токен у localStorage.
- **Lazy loading**: використання `React.lazy` для підвантаження великих секцій (Dashboard) без збільшення початкового bundle.

### 4.6 Zod
- **Schema** в `useTasks`:
```ts
const taskSchema = z.object({
  id: z.string().uuid(),
  title: z.string().min(1),
  status: z.enum(['TODO','IN_PROGRESS','DONE']),
  priority: z.enum(['LOW','MEDIUM','HIGH']).optional(),
});
```
- **Валідація** запиту до бекенда (Fastify/Express) дозволяє виявляти некоректні дані ще до запису в БД.
- **Type inference** забезпечує тип `Task = z.infer<typeof taskSchema>`.

### 4.7 CI/CD у Vercel
- **Automatic preview builds** після кожного PR → миттєва перевірка UI.
- **Edge Functions** (наприклад, middleware для авторизації) працюють у безсерверному режимі, з низькою латентністю.
- **Zero‑config deployments**: Vercel автоматично розпізнає `package.json` scripts (`npm run build`) і розгортає `dist/`.

## 6. Висновок

Після комплексного аналізу технологічного стеку **React + TypeScript + Vite + Tailwind CSS + Prisma + React‑Router + Zod** на підставі **продуктивності розробки, швидкості** (HMR <50 мс), **масштабованості** (можливість підключення мікросервісів) та **забезпечення типобезпеки** у всіх шарі, ми можемо стверджувати, що обраний стек забезпечує оптимальний баланс між швидкістю розробки, якістю коду та можливістю подальшого розширення. Порівняння з Vue, Angular, Webpack, Sass, TypeORM, Joi показало, що альтернативи не дають суттєвих переваг у контексті даного проєкту, а лише ускладнюють процес розгортання та підтримки.

Наступний етап – реалізація архітектури, налаштування CI/CD, написання тест‑сьюту та впровадження функціональних модулів згідно вимог, сформульованих у першому розділі.
