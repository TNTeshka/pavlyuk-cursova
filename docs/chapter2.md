# Огляд технологій та засобів розробки

## Вступ
У сучасному розробленому середовищі вимоги до веб‑застосунків зростають: потрібна швидка розробка, висока продуктивність, зручність підтримки, масштабованість та можливість адаптації під різні пристрої і теми. У цьому розділі детально розглядається технологічний стек, використаний у створенні системи управління задачами «Task Manager», а також аналізуються альтернативи, які можуть служити заміною. Порівняння допоможе підтвердити доцільність вибору інструментів, а також виявити їх переваги у контексті нашого проєкту.

## Глибокий розбір ключових технологій

### React
React залишається провідним інструментом для створення реактивних інтерфейсів. У нашому проєкті він використовується для реалізації **SidebarLayout**, **KanbanBoard** та **TaskModal**. Компонентна модель дозволяє ізолювати логіку управління станом (за допомогою хук `useState`, `useMemo`) та розбити інтерфейс на повторно використовувані частини (`Button`, `Header`). React‑фокус на **immutability** спрощує відстеження змін стану, що важливо при роботі з великим набором задач.

#### Переваги у нашому контексті
- **Легка інтеграція з TypeScript**: типи `Task`, `TaskStatus`, `TaskPriority` гарантують коректність даних.
- **Спільнота та бібліотеки**: `react-router-dom` для навігації, `react-query` (можна додати) для кешування запитів.
- **Підтримка SSR (сервер‑сайд рендеринг)**: потенційно можна розширити для SEO у майбутньому.

#### Альтернативи та їх недоліки
- **Vue**: має схожу реактивну систему, проте екосистема для бекенда (наприклад, Vite + Vue) ще не так поширена в нашій команії.
- **Angular**: важка крива навчання, вимагає TypeScript, але великий розмір фреймворку не підходить для мікросервісів.

### TypeScript
Використання TypeScript у проєкті дозволяє знизити кількість помилок рантайму, особливо при роботі з ORM‑моделями Prisma. За допомогою інтерфейсів (`Task`, `Group`) ми формалізуємо контракт між фронтендом та бекендом, що спрощує тестування та рефакторинг.

#### Порівняння з JavaScript
- **JavaScript (ES2022)**: без типізації, швидший старт, але під час розробки часто виявляються типові помилки (наприклад, передача `undefined` в запит).
- **Flow**: альтернативний типізатор, менш активно підтримується, менше інтеграції з бібліотеками.

### Vite
Vite замінив традиційний Webpack у нашому стеку, що суттєво прискорило процес розробки. Завдяки нативному ES‑модульному імпорту, сервер Vite миттєво підвантажує лише змінені модулі, а HMR працює без затримок.

#### Переваги в порівнянні з CRA/Webpack
- **Швидкість**: старт сервера за секунди, а не хвилини.
- **Конфігурація «з коробки»**: просте підключення Tailwind CSS через `postcss.config.js`.
- **Модульність**: легке додавання плагінів, наприклад `vite-plugin-react`.

#### Чому не Webpack?
- Webpack потребує значного конфігураційного коду, що ускладнює підтримку.
- Після оновлення залежностей часто виникають конфлікти.

### Tailwind CSS
Tailwind дозволив нам швидко створювати **responsive** інтерфейс без написання власних CSS‑класів. Використовуючи утилітарні класи, ми досягли узгодженого стилю між різними компонентами, включаючи **Header**, **Sidebar**, **KanbanBoard**.

#### Переваги над Sass/Bootstrap
- **Zero‑runtime CSS**: лише ті класи, які використовуються, залишаються у фінальному CSS‑файлі.
- **Тема‑aware**: легко змінюємо `data-theme` атрибут, а Tailwind автоматично застосовує інші колірні змінні.
- **Незалежність від готових компонентів**: створюємо UI‑шаблони під наші вимоги.

#### Недоліки альтернатив
- **Sass** вимагає писати власні правила, що збільшує розмір CSS.
- **Bootstrap** накладає готову структуру, що обмежує кастомізацію.

### Prisma
Prisma використовується в бекенді (Node.js) для взаємодії з базою даних PostgreSQL. ORM генерує типи для моделей (`User`, `Group`, `Task`) і дозволяє писати **type‑safe** запити, що зменшує ризик SQL‑ін’єкцій.

#### Переваги у порівнянні з TypeORM/Sequelize
- **Тезаурус типів**: автоматична генерація типів, що працює зі TypeScript.
- **Міграції**: простий CLI для створення та застосування міграцій.
- **Продуктивність**: Prisma оптимізує запити, використовуючи prepared statements.

#### Чому не TypeORM?
- TypeORM часто має проблеми з синхронізацією схем у великих проєктах.
- Менша стабільність при використанні нових версій TypeScript.

### React‑Router
Для навігації в SPA ми використовуємо `react-router-dom` (v6). Це дозволяє створювати **protected routes**, які перевіряють аутентифікацію користувача, а також передавати параметри (`/groups/:groupId/tasks`).

#### Переваги над Reach Router
- **Більша підтримка в спільноті** і активні оновлення.
- **Гнучка система вкладених роутів** і перехідних анімацій.

### Zod
Zod використовується для валідації вхідних даних у бекенд‑контролерах та у фронтенд‑хук‑ах (`useTasks`). Переваги Zod включають **автоматичну типізацію** (з використанням інтерфейсу `z.infer`) та **вбудовану підтримку async‑валидації**.

#### Порівняння з Joi/Yup
- Zod **не вимагає схем у JSON**, а використовує TypeScript‑подібний синтаксис.
- Joi та Yup часто потребують трансляції типів, що додає надмірність коду.

## Висновок
Після комплексного аналізу стало очевидним, що обраний технологічний стек **React + TypeScript + Vite + Tailwind CSS + Prisma + React‑Router + Zod** найкраще відповідає вимогам нашого проєкту:

1. **Продуктивність розробки** – Vite забезпечує швидкий цикл «змінити‑переглянути», а TypeScript скорочує час на відлагодження.
2. **Гнучкість UI** – Tailwind дозволяє швидко створювати чистий, адаптивний інтерфейс, включаючи темний/світлий режим з градієнтним фоном.
3. **Масштабованість та типобезпечність** – Prisma і Zod гарантують, що дані, які передаються між фронтендом і бекендом, відповідають строгим типам, що спрощує розширення функціоналу у майбутньому.
4. **Легка інтеграція та підтримка** – всі інструменти активно підтримуються спільнотою, мають добре задокументовані API та готові плагіни, що значно знижує ризики у процесі розробки.
5. **Вартісна ефективність** – більшість інструментів безкоштовні або мають відкриті ліцензії, а хостинг на Vercel/Netlify дозволяє розгорнути проект без додаткових витрат.

Таким чином, технологічний стек, обраний для реалізації системи управління задачами, є оптимальним рішенням як з точки зору технічної придатності, так і з точки зору економічної ефективності. Він забезпечує стабільну основу для подальшого розвитку, розширення функціоналу та підтримки користувачів у довгостроковій перспективі.
