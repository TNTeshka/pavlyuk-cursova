# 3.2. Клієнтська частина (Frontend)

**У процесі розробки інтерфейсу користувача системи управління задачами** було обрано модульно‑шарову архітектуру, що забезпечує чітке розділення відповідальності між UI‑компонентами, станом додатку та сервісним шаром, який взаємодіє з бекендом через REST‑API та WebSocket. Така організація дозволяє **ізолювати логіку відображення** від **бізнес‑логіки**, спрощує тестування, підвищує масштабованість і робить можливим незалежне розгортання клієнтської частини.

## 3.2.1. Архітектура та налаштування UI‑середовища

### Ініціалізація проєкту
Клієнтська частина ініціалізована за допомогою **Vite** (версія 6.x) у поєднанні з **React 19** та **TypeScript**. Vite забезпечує **швидкий старт** і **миттєве оновлення** під час розробки, а TypeScript гарантує сувору типізацію всіх компонентів. У файлі `vite.config.ts` налаштовано псевдоніми (`@/components`, `@/hooks` тощо) для зручного імпорту, а також вбудовано **Tailwind CSS** як утилітарний CSS‑фреймворк.

### Tailwind CSS та темізація
Tailwind CSS обрано за його **утилітарну природу**, що дозволяє будувати стилистично узгоджені інтерфейси без створення великої кількості кастомних CSS‑класи. Файл `tailwind.config.cjs` містить розширені кольорові палетти, кастомні шрифти та параметри «dark mode», який активується на основі системних налаштувань користувача (`media`) або вручну через перемикач у UI. Приклад конфігурації теми:
```js
module.exports = {
  darkMode: 'class', // дозволяє перемикати теми класом
  theme: {
    extend: {
      colors: {
        primary: '#2563eb',
        secondary: '#1e3a8a',
        accent: '#f43f5e',
      },
    },
  },
  plugins: [],
};
```
Це забезпечує **однорідність візуального оформлення** на всіх сторінках і полегшує підтримку **dark/light** режиму.

### Структура компонентів
Компонентна архітектура побудована за принципом **feature‑sliced** – кожна функціональна ділянка (наприклад, `Header`, `KanbanBoard`, `GroupDashboard`, `TaskModal`, `UpdateTaskModal`) розташована у власній папці, містить **React‑компонент**, **type‑definition**, **тести** і **стилі** (Tailwind‑класи). Це полегшує **масштабування**: новий модуль можна додати без змін у глобальній структурі.

### Global Layout та роутинг
Маршрутизація здійснюється за допомогою `react-router-dom` (версія 6.x). Головний layout (`src/App.tsx`) містить глобальні елементи‑обгортки: навігаційний `Header`, `SidebarLayout` та `Outlet` для динамічних сторінок. Файл `src/routes.tsx` описує дерево маршрутів, включаючи захищені маршрути (через `RequireAuth`), які доступні лише після автентифікації.

## 3.2.2. Управління станом та інтеграція з API

### Zustand – глобальний стан
Для управління глобальним станом використано **Zustand**. На відміну від Redux, Zustand не вимагає створення великих boilerplate‑кодів і дозволяє **записувати стан у вигляді звичайних JavaScript‑об’єктів**. У проєкті створено три основних стора:
- `useTaskStore` – зберігає колекцію задач, їх статуси, пріоритети та метадані.
- `useGroupStore` – інформація про групи, членів та ролі.
- `useUserStore` – дані про аутентифікованого користувача, JWT‑токен та налаштування теми.

### React Query – кешування та асинхронні запити
Для взаємодії з бекендом використано **React Query** (версія 5.x). React Query автоматично кешує відповіді, оновлює дані після мутацій та забезпечує **stale‑while‑revalidate**‑модель. Приклад запиту списку задач:
```ts
const { data: tasks, isLoading } = useQuery(['tasks', filter], () =>
  api.get('/tasks', { params: filter }).then(res => res.data)
);
```
Мутації (створення, оновлення, видалення) виконуються через `useMutation`, після чого викликається `queryClient.invalidateQueries(['tasks'])`, що оновлює кеш.

### HTTP‑клієнт (fetch)
Для здійснення запитів використано нативний `fetch` у обгортці `api.ts`, яка автоматично додає `Authorization: Bearer <accessToken>` до заголовків і обробляє помилки (401 → перенаправлення на сторінку входу).

### WebSocket‑інтеграція
Для синхронізації стану Kanban‑дошки у реальному часі використано **Socket.io‑client**. Після успішного входу користувач ініціалізує з’єднання, передаючи JWT у `auth`‑полі. Події `task:created`, `task:updated`, `task:deleted` підписуються у `src/socket.ts` і оновлюють відповідні Zustand‑стори без необхідності повторних запитів до серверу.
```ts
import { io } from 'socket.io-client';
import { useTaskStore } from './store/taskStore';

export function initSocket(accessToken: string) {
  const socket = io(import.meta.env.VITE_API_URL, { auth: { token: accessToken } });
  socket.on('task:created', payload => useTaskStore.getState().addTask(payload.task));
  socket.on('task:updated', payload => useTaskStore.getState().updateTask(payload.task));
  socket.on('task:deleted', payload => useTaskStore.getState().removeTask(payload.taskId));
  return socket;
}
```
Таке рішення гарантує **мінімальну затримку** між операцією користувача та відображенням результату на всіх підключених клієнтах.

## 3.2.3. Користувацький досвід та адаптивність

### Роль‑орієнтований UI
Для задоволення різних прав доступу розроблено **role‑based rendering**: у компоненті `Header` та в меню відображаються лише ті елементи, які дозволені ролі поточного користувача. Це реалізовано через контекст `useUserStore`, який надає поле `role`; у JSX використовується умовний рендеринг (`{role === 'ADMIN' && <AdminPanel />}`). Таким чином, **адміністратори**, **менеджери** та **звичайні користувачі** бачать інтерфейси, адаптовані до їхніх прав.

### Швидке створення та фільтрація задач
На панелі `KanbanBoard` реалізовано **швидке додавання** задач через інлайн‑введення у кожному стовпці (TODO, IN_PROGRESS, DONE). Фільтрація здійснюється на боці клієнта за допомогою `useMemo`, що споживає глобальний стан `searchTerm` і `quickFilter` (високий пріоритет, прострочені, на сьогодні). Змінюючи ці параметри, користувач миттєво бачить відфільтрований список без зайвих запитів до серверу.

### Адаптивність та responsive design
Tailwind CSS забезпечує **responsive utilities** (`sm:`, `md:`, `lg:`), що дозволяє легко створювати **мобільну версію** Kanban‑дошки: у вузькому вікні колонки вертикально зсуваються, а у широкому – розташовуються у триколонковому вигляді. Використання `flex` та `grid` у компонентах гарантує, що UI коректно адаптується до різних розмірів екранів.

### Темізація (dark/light mode)
Користувач може перемикати тему через кнопку у `Header`. Перемикач змінює клас `dark` на кореневому елементі `<html>`, що активує відповідні Tailwind‑класи (`bg-gray-900`, `text-white` тощо). Обрану тему зберігає `localStorage`, а під час ініціалізації застосунок зчитує її та встановлює.

## Висновки
Клієнтська частина системи управління задачами реалізована на **React 19 + TypeScript**, що забезпечує **продуктивність** та **дефінітивну типізацію**. Використання **Vite** спрощує процес розробки, а **Tailwind CSS** дозволяє швидко створювати адаптивний, тематизований інтерфейс. **Zustand** і **React Query** у поєднанні забезпечують **ефективне управління станом** та **кешування даних**, мінімізуючи кількість запитів до серверу. **Socket.io** гарантує **мгновену синхронізацію** змін у реальному часі, що критично для Kanban‑дошки. Завдяки **role‑based rendering** інтерфейс адаптується під різні ролі користувачів, а система темізації підвищує доступність та комфорт користування. У підсумку, фронтенд відповідає вимогам щодо **масштабованості**, **зручності розробки**, **підтримки** та **високої продуктивності**, створюючи надійну основу для подальшої інтеграції з бекенд‑частиною.
